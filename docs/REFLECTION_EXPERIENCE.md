# Evo-PentestRAG 反思机制与经验总结

## 概述

本文档总结了 Evo-PentestRAG 系统在实战中积累的反思经验和优化策略。这些经验来自于对攻击日志的深度分析，以及基于"System 2思考"的自适应查询优化机制。

---

## 一、系统演进路线图

### 第一步：数据"炼金" (From Logs to Dataset)

**目标**：利用现有的日志，自动化生成用于训练"价值判别器"的数据集。这是系统"自进化"的起点。

#### 执行策略

- **输入**：`logs/cai_*.jsonl` 攻击日志文件
- **工具**：`processors/auto_labeler_v2.py`（基于后见之明的自动标注器）
- **执行动作**：
  1. **解析轨迹**：将日志按 `Trace ID` 分组
  2. **提取三元组**：提取 `(Query, Document, Execution_Result)`
  3. **自动打分 (Hindsight Labeling)**：
     - 如果后续执行结果含 `uid=0`, `success`, `flag{` → **Label: 1.0 (正样本)**
     - 如果后续执行结果含 `error`, `not found`, 或立即换词重搜 → **Label: 0.0 (负样本)**
  4. **清洗**：丢弃无法判断的中性样本，去除重复数据
- **产出**：`processors/dataset_train.jsonl`

#### 关键代码位置
- `processors/auto_labeler_v2.py` - HER经验提取器
- `processors/dataset.py` - 数据集加载和处理

---

### 第二步：核心模型训练 (Training the Utility Scorer)

**目标**：训练一个能判断"知识是否有用"的大脑。这是论文的核心 Contribution 之一（Functional Alignment）。

#### 训练配置

- **输入**：`dataset_train.jsonl`
- **架构**：**Cross-Encoder** (基于 `ms-marco-MiniLM-L-6-v2`)
- **执行动作**：
  1. 加载数据集
  2. 使用 `BCELoss` 或 `RankingLoss` 进行微调
  3. 训练 3-5 个 Epoch（防止过拟合，使用 Early Stopping）
- **产出**：模型权重文件存储在训练目录

#### 训练结果分析

根据 `processors/TRAINING_REPORT.md` 的记录：

- **总样本数**: 395
- **训练集**: 316 (80%)
- **测试集**: 79 (20%)
- **Loss下降**: 61.4 → 0.38 (下降99.4%)

#### 关键代码位置
- `processors/trainer.py` - 模型训练器
- `processors/train_balanced.py` - 平衡数据集训练
- `processors/evaluator.py` - 模型评估器

---

### 第三步：构建"反思"模块 (System 2 Reflection)

**目标**：不仅仅是给知识打分，还要从失败中总结出"文字版经验"。

#### 反思机制架构

1. **失败检测器 (Failure Detector)**
   - 识别检索失败的模式
   - 触发条件：Top-5 文档经 LLM 判断"无法形成攻击方案"

2. **多维度诊断器 (Reflection Diagnoser)**
   - **Query 漂移**：搜索词与实际需求不匹配
   - **粒度失衡**：内容过于宽泛或过于具体
   - **版本不匹配**：目标版本与文档版本差异
   - **领域偏移**：搜索内容与目标环境不符

3. **查询改写器 (Query Rewriter)**
   - 根据反思结论，自动生成 3 个不同维度的变体 Query
   - 采用多策略改写：扩展、聚焦、替换、结构化

#### 经验总结策略

从"先失败、后成功"的日志片段中提取策略：

```python
# 示例经验提取
{
  "场景": "WebLogic 返回 403 Forbidden",
  "失败原因": "使用明文 Payload 被 WAF 拦截",
  "成功策略": "优先尝试 XML 编码绕过",
  "适用条件": "中间件为 WebLogic 且存在 WAF"
}
```

#### 关键代码位置
- `processors/failure_detector.py` - 失败检测器
- `processors/reflection_diagnoser.py` - 反思诊断器
- `processors/query_rewriter.py` - 查询改写器
- `processors/adaptive_retriever.py` - 自适应检索系统（集成所有组件）

## 二、RAG 深度优化四步法

### 第一步：构建"后果驱动"的自动标注流水线

**核心思想**：「Functional Alignment」- 传统的 RAG 标注看"相关性"，我们的 RAG 标注看"是否有助于拿到 Shell"。

#### 轨迹回溯机制

- **成功标准**：定位所有导致 `success: true` 或获得高价值反馈的代码执行动作
- **效用评价函数** $R(d)$：
  - 如果在搜索到文档 $d$ 后的 3 个步骤内，成功执行了与 $d$ 内容高度相关的 Exploit → 高分 (1.0)
  - 如果阅读了 $d$ 但接下来的动作依然是重试或报错 → 低分 (-1.0)

#### 数据集特点

*这是全球首个基于"实战后果"生成的渗透知识搜索评估数据集*

---

### 第二步：训练"攻击价值"重排序模型

**核心思想**：解决向量检索"只认词索引，不认攻击逻辑"的问题。

#### 模型架构

- **基础检索**：向量检索 Top-50
- **重排序**：Cross-Encoder Reranker
- **训练目标**：输入「靶机指纹 + 已执行命令」和「文档片段」，预测该文档能为下一步攻击提供有效指令的概率

#### 负采样机制

专门从那些"语义非常接近但版本号不对"或"看似有用实则过时"的文档中进行难负采样（Hard Negative Mining）。

**示例**：当靶机是 CouchDB 3.x 时，模型应学会将 2.x 的 Exploit 文档排在后面，即便它们的语义非常相似。

---

### 第三步：自适应查询自反思

**核心思想**：「System 2 Reflection」- 让 RAG 具有发现自己"搜错了"的能力。

#### 触发机制

**搜索失败触发器**：当检索回来的 Top-5 文档经过 LLM 判断"无法形成攻击方案"时，触发反思逻辑。

#### 多维度诊断

1. **Query 漂移**："我搜的是 CVE 编号，但可能还没分配，应该搜产品名+特征词"
2. **粒度失衡**："当前内容太泛了，需要针对某个特定的 API 进行检索"
3. **版本不匹配**："文档版本与目标版本不符"
4. **领域偏移**："搜索内容与目标环境不符"

#### 查询改写策略

根据反思结论，自动生成 3 个不同维度的变体 Query 重新检索，实现"探索-利用"平衡。

---

### 第四步：技术密度过滤与多模态提纯

**核心思想**：解决爬虫语料中的"噪音问题"（SEO 文章、图片缺失导致的断头文）。

#### TDS (Technical Density Score) 评估

基于 LLM 的轻量级评分器，对爬取的公众号文章进行"干货度"打分：
- **Payload 密度**：实际可用的攻击代码比例
- **配置命令占比**：可直接执行的命令数量
- **逻辑推导连贯性**：攻击链路的完整性

#### 跨模态补全

利用 VLM（如 GPT-4o 或本地模型）解析文章中由于反爬没抓到的图片：
- 将"如下图所示"转变为具体的文字描述
- 例如："在管理后台点击 Settings → Security 选项卡"

*创新点：通过多模态知识对齐，消除渗透知识库中的单模态盲区*

## 三、知识库扩容与提纯

### 源头发现策略

- **RSSHub 监控**：利用 RSSHub 监控安全社区引用，发现新公众号
- **质量筛选**：计算 TDS (技术密度分)，只爬取硬核文章
- **多模态清洗**：爬取文章时，检测 `<img>` 标签 → OCR/VLM 转文字 → 插入正文

### 去重与降权

- 新爬取的文章先经过 Utility Scorer 打分
- 如果分低或是重复内容，直接丢弃或降权
- 更新后的数据推送至向量数据库 (RAGFlow)

## 五、实验与评估指标

### 实验设计

- **实验组 (Ours)**：完整版 Evo-PentestRAG
- **对照组 (Baseline)**：只有基础 RAG 的 Agent

### 关键指标

1. **ASR (Attack Success Rate)**：攻击成功率
2. **Avg. Steps**：拿到 Shell 的平均步数（越少越好，证明更聪明）
3. **Noise Robustness**：往库里塞 5000 条垃圾数据，看谁受影响更小
   - 预期：我们的模型应该能把垃圾排在后面

## 六、核心创新点总结

### 1. Functional Alignment（功能对齐）
不看语义相关性，看实战价值。首个基于攻击后果的知识评估体系。

### 2. System 2 Reflection（系统二反思）
让 RAG 具备自我纠错能力，从失败中学习。

### 3. Hard Negative Mining（难负样本挖掘）
专门训练模型识别"看似有用实则无用"的知识。

### 4. Multi-Modal Knowledge Alignment（多模态知识对齐）
通过 VLM 补全图片信息，消除单模态盲区。

### 5. Adaptive Query Rewriting（自适应查询改写）
探索-利用平衡的渗透词检索策略。
