# RefPenTest 项目方案：基于真实日志的深度重构

## 一、真实日志深度解析

### 1.1 日志真实结构

每个"轮次"由三类记录交替构成：

```
[assistant_message]      ← 当前轮次Agent回复（content通常为空壳）
[API请求对象]            ← 包含完整messages历史（含所有历史tool_call+tool_result）
[chat.completion响应]    ← 包含本轮实际tool_call决策
```

这意味着每个 API请求对象 里天然携带了**完整的上下文链**，反思引擎可以直接从中还原完整攻击决策链。这是本日志格式的一个极其有价值的特征。

### 1.3 三个改变设计方向的关键发现

**发现①：RAG被调用了，但根本没被用**

第一次RAG搜索发生在Line 70，搜到了CVE-2017-10271的GitHub README（相似度0.817），但在随后的Line 75，Agent直接调用`execute_code`**自己从头写了exploit脚本**，与RAG返回内容几乎没有关系。这完美验证了"相似度≠真正有用"的核心命题。这是本项目最重要的实证。

**发现②：反弹Shell失败的根本原因是网络拓扑盲点**

Agent理解了"获取反弹Shell需要监听IP"，也正确获取了本机IP 172.18.79.129，但**未意识到目标是127.0.0.1（loopback），从loopback环境向外发起TCP连接到172.18.79.129是网络不可达的**。这不是exploit问题，而是网络拓扑判断失误——纯粹靠相似度检索的RAG无法捕捉这种"场景理解失败"。

**发现③：工具使用存在系统性问题**

gobuster参数调用错误（缺必填参数）、dirb未预先检查是否安装，这说明Agent的Procedural知识有明显缺口，但现有RAG没有为"工具使用前提检查"建立对应的语料。

## 二、关于三个核心问题的深入判断

### 2.1 是否引入大模型判断？

**结论：必须引入，且是本项目的核心技术点，但要精准定位LLM的职责边界。**

从日志可以看到，判断"Agent是否真正用了RAG内容"这件事，是无法用规则完成的。例如本次日志中，RAG返回了CVE-2017-10271的GitHub描述，Agent随后写出的exploit代码在逻辑上是"相关的"，但实际上是Agent自有知识生成的，和RAG内容没有行为因果关系。这种判断**只有LLM能做**。

LLM在本项目中的三个精准职责如下：

**职责1：RAG行为因果判定（最核心）**

```
输入：[RAG返回内容] + [RAG查询后N步内的tool_calls和tool_results]
任务：判断Agent的后续行为是否真正采纳了RAG内容中的关键信息
      输出：adopted / ignored / partial_adopted + 判断依据
```

本次日志的判定结果就是`ignored`——RAG返回了"使用XMLDecoder反序列化漏洞"的说明，但Agent后续的exploit代码是自己生成的，未引用RAG中的具体payload格式或代码片段。

**职责2：失败根因归类（渗透测试专项分类器）**

```
输入：[失败的tool_call] + [tool_result错误信息] + [上下文攻击目标]
任务：从渗透测试视角归类失败原因
      分类法（五维根因框架，而非枚举列表）：
      - 维度一 ENV（执行环境问题）：失败原因在执行环境本身，与攻击策略无关
      - 维度二 INV（调用方式问题）：工具/命令调用方式有误，修正调用即可成功
      - 维度三 DEF（目标防御问题）：目标存在主动或被动防御机制阻止了攻击
      - 维度四 INT（情报/认知问题）：对目标或环境的认知不准确/不完整
      - 维度五 EFF（执行效果问题）：命令被触发但效果与预期不符
      LLM依据此框架对任意工具的任意失败做根因归类，无需预先列举具体类型
```

**职责3：经验条目生成（结构化输出）**

```
输入：一个完整的攻击阶段（含所有events）
任务：按四层知识模型生成结构化经验JSON
      注意：LLM不做入库决策，只做内容提取，入库由规则引擎决定
```

不需要LLM的部分（用规则/代码完成更可靠）：
- RAG查询识别（识别`make_kb_search`调用）
- 确定性失败预判（以下三条100%可靠，与工具无关）：
    return_code=127 → ENV-BINARY_MISSING（shell通用约定，命令不存在）
    timed_out=true  → ENV-TIMEOUT（工具自报告字段，直接读取）
    return_code=126 → ENV-EXECUTE_PERMISSION_DENIED
- 经验是否与已有条目重复（向量相似度）
- 时间统计、计数统计

### 2.2 是否需要训练模型？

**结论：近期不需要训练，中期可考虑微调，但不是创新点所在。**

训练一个专用的渗透测试意图分类模型确实有价值，但存在以下问题：

其一，渗透测试日志数据量极其有限，CTF/靶场日志是稀缺资源，冷启动困难。其二，用大模型做few-shot分类，在准确率上已经足够满足本项目的反思需求，性价比更高。其三，本项目的真正创新点在**反思机制的设计**，而不在基础NLP分类能力。

中期可考虑的方向是：积累足够多的经验条目后，用经验库数据对Embedding模型做**领域微调**，使"渗透测试专项检索"的语义理解优于通用Embedding，让"反弹Shell loopback场景"和"反弹Shell正常场景"在向量空间上有明确区分。这个贡献可以作为论文的一个子实验。

### 2.3 要做专业"渗透测试"反思RAG，不是通用反思？

**这是本项目最核心的差异化设计原则。** 从本次日志中，我们可以清晰看到"通用反思"和"渗透测试专项反思"的本质区别：

通用反思会记录"bash命令执行失败"，专业渗透测试反思必须记录"在loopback靶机环境下，反弹Shell无法通过TCP/IP回连外部IP，应改用bind shell或本地提权策略"。通用反思会记录"工具调用失败"，专业渗透测试反思必须区分是"工具未安装"、"工具参数错误"、"目标拦截"、"版本不兼容"四种完全不同的失败，因为它们的应对策略完全不同。

这个原则将贯穿整个反思体系的设计。

## 三、重构后的完整项目流程

### 3.1 日志解析层

日志格式本质：**每个API请求对象携带完整的历史消息链**，因此解析策略是"以最新API请求对象为一个解析单元"。

```
解析单元 = 一个完整的API请求对象（UNKNOWN行）

从中提取：
1. system_prompt → 提取Agent角色定义和能力边界
2. messages[-1] → 当前轮次的输入（最新tool_result）
3. 向前扫描messages → 找到本轮的tool_call决策
4. 向前扫描messages → 找到上一次make_kb_search的结果（如有）
5. 对应的chat.completion响应 → 下一步的tool_call决策

全量扫描逻辑：
- 识别 make_kb_search（或任何等价的知识检索调用）→ 标记RAG查询事件，
  记录查询内容和时间戳，建立与后续N=3步的时间窗口关联
- 识别 execute_code（或任何等价的代码写入/执行调用）→ 标记代码执行事件，
  提取代码内容和执行结果，通过文件名/标识符关联后续的脚本执行调用
- 识别所有其他 tool_call（tool_name原样保留，不做分类）→ 标记为工具执行事件，
  全量无差别进入反思分析流，工具语义判断交给Layer 1
  
非RAG支撑事件标记：
  每个 tool_call 新增字段 has_rag_support: true/false/partial
  判断逻辑：该 tool_call 前N步内（建议N=3）是否有 make_kb_search 且判定为 adopted/partial
  → has_rag_support=false 的 tool_call 是"知识缺口检测"的核心输入
```

针对你日志的特殊处理：

```
特殊字段：usage.prompt_tokens
  作用：prompt_tokens持续增长说明上下文在累积
  反思价值：如果prompt_tokens增长但无效果，说明上下文膨胀，无有效知识注入
```

### R-01：RAG行为因果判定规则（核心创新）

```
触发：任何 make_kb_search 调用后
评判维度：
  A. 内容相关性（LLM判定）：RAG返回内容是否真正回答了查询意图？
     示例：本次RAG#1查询意图是"找exploit"，返回的是README说明文字 → 相关但不充分
  
  B. 行为采纳度（LLM判定）：后续3步tool_call是否采纳了RAG中的具体信息？
     判断依据：RAG中有具体payload/命令/代码 且 后续命令与之重合 → adopted
               Agent自写代码/命令与RAG内容无关 → ignored
     本次：RAG#1 → ignored（相似度0.817，但行为采纳度≈0）
  采纳度三级定义：
  Level 3 - 直接引用（Direct Adoption）：
    后续tool_call的参数/代码中包含RAG内容中的具体字符串（命令/代码片段/路径/端点）
    权重：1.0
    
  Level 2 - 参考改写（Informed Adaptation）：
    后续行为的技术路线与RAG内容一致，但具体命令/代码由Agent自行生成
    判定依据：技术方法重合（如RAG提到XML反序列化，代码也用XML反序列化）
    权重：0.6
    
  Level 1 - 思路启发（Conceptual Influence）：
    RAG内容影响了攻击方向选择，但具体技术路线不同
    判定依据：RAG后出现了漏洞/技术的方向切换
    权重：0.3
    
  Level 0 - 未采纳（Ignored）：
    后续行为与RAG内容无任何关联
    权重：0.0

行为采纳度（BAR）计算：
  BAR = 该session中所有RAG调用的加权采纳级别之和 / RAG调用总次数
  取值范围：[0.0, 1.0]
  这个指标作为论文的核心实验变量，对比有/无反思经验库时的BAR变化
  
  
  C. 结果贡献度（规则判定）：该RAG调用发生的阶段最终是否成功？
  
综合判定 → 写入经验：
  "make_kb_search query=CVE-2017-10271 exploit，returned README-level content，
   agent ignored and self-wrote exploit → KB content insufficient for exploit-level task"
  
  → 触发：知识库缺口记录（需要的是可用的exploit代码，而非漏洞描述）
  
补充说明：
  无 make_kb_search 触发的 tool_call 不进入 R-01，
  但必须进入 R-02（失败分类）或 R-03（成功提取）处理流，
  且统一标记 has_rag_support=false，触发 R-07 知识缺口检测。
```

### R-02：渗透测试专项失败分类规则

```
【失败根因五维分类法】

维度一：执行环境问题（Execution Environment）
  定义：失败的原因在于执行环境本身，与攻击策略无关
  特征：换一个正确配置的环境就能成功
  子维度（这里可以列举，因为环境问题是有限的）：
    ENV-BINARY_MISSING    工具/可执行文件不存在
    ENV-DEPENDENCY_MISSING 工具的运行时依赖缺失
    ENV-PERMISSION         当前进程权限不足以执行该操作
    ENV-TIMEOUT            执行超时（网络/计算资源）
    ENV-RESOURCE           内存/磁盘/文件描述符等资源不足

维度二：调用方式问题（Invocation）
  定义：失败的原因在于工具/命令的调用方式本身有误，与目标无关
  特征：修正调用方式就能成功
  子维度：
    INV-WRONG_ARGS         参数错误/缺失必填参数
    INV-WRONG_SYNTAX       命令语法错误
    INV-WRONG_SEQUENCE     前置步骤缺失（先决条件未满足）
    INV-WRONG_CONFIG       配置/选项设置有误

维度三：目标防御问题（Target Defense）
  定义：目标存在主动或被动的防御机制阻止了攻击
  特征：绕过防御或换方法才能成功
  子维度：
    DEF-AUTHENTICATION     需要认证但未提供有效凭据
    DEF-AUTHORIZATION      已认证但权限不足
    DEF-ACTIVE_BLOCKING    WAF/IDS/防火墙等主动拦截
    DEF-RATE_LIMITING      速率限制/请求频率控制
    DEF-PATCHED            目标已修补该漏洞
    DEF-MONITORING_DETECT  被监控系统检测（触发告警后封锁）

维度四：情报/认知问题（Intelligence Gap）
  定义：失败的原因在于对目标或环境的认知不准确/不完整
  特征：补充正确信息后才能成功
  子维度：
    INT-WRONG_VERSION      版本判断错误（漏洞不适用于该版本）
    INT-WRONG_TOPOLOGY     网络拓扑理解错误（loopback/内网/NAT等）
    INT-INCOMPLETE_RECON   侦察不充分，关键信息未收集
    INT-WRONG_ASSUMPTION   错误假设目标存在某漏洞/服务/配置
    INT-MISSING_CONTEXT    缺少利用漏洞所需的上下文信息

维度五：执行效果问题（Execution Effect）
  定义：命令/漏洞被成功触发，但效果与预期不符
  特征：需要调整后渗透策略
  子维度：
    EFF-BLIND_EXECUTION    执行成功但无回显（盲RCE/盲注等）
    EFF-PARTIAL_SUCCESS    部分成功（只触发了攻击的一部分）
    EFF-OUTPUT_LOST        输出存在但无法获取（权限/路径问题）
    EFF-UNSTABLE           间歇性成功（竞争条件/不稳定漏洞）
```

### R-08：经验库的更新融合机制

**攻击链融合的触发逻辑应该分三层：**

第一层是横向融合——同一漏洞在不同环境中的正向经验合并。例如 CVE-2017-10271 在三个不同 WebLogic 版本上都成功利用，每次的有效端点可能略有不同，这时需要把三条 Factual 经验融合为一条带"版本-端点"映射表的 Conceptual 经验，并标注"适用版本范围"和"已知无效端点"。

第二层是纵向融合——同一漏洞的正向和负向经验合并成完整攻击手册。例如有三条关于 CVE-2017-10271 的条目：一条是成功的 exploit 代码、一条是"loopback 环境不能用反弹 shell"的负向经验、一条是"盲RCE需要OOB获取输出"的补充经验。这三条应该融合为一个完整的"CVE-2017-10271 利用指南"，包含正确做法、已知陷阱、环境前提检查清单。

第三层是通用化提升——当某个攻击链被验证在多种中间件/框架上都适用时，应从漏洞特定经验升级为攻击模式经验。例如"Java 反序列化漏洞通用利用思路"可以从多条 WebLogic/JBoss/Shiro 的 Factual 经验中提炼出来。

## 五、面向顶会的创新体系定位

这是整个项目最需要明确的部分，也是与已有RAG反思工作的本质差异化所在。

### 5.1 现有工作的局限性

目前学术界的RAG反思工作（SELF-RAG, CRAG, Self-Reflective RAG 等）存在三个共同局限：

第一，评估停留在文本层面，以BLEU/ROUGE或人工标注的相关性分数作为评估指标，无法衡量"检索内容是否真正影响了行为"。

第二，反思对象是静态的QA任务，不涉及多步骤决策链和行为序列。

第三，没有领域专项知识结构，经验以平铺向量存储，无法区分"需要事实"还是"需要决策规则"。

### 5.2 本项目的三个核心创新点

**创新点 I：行为因果闭环评估**

这是本项目最强的顶会级创新。现有RAG评估的终点是"检索质量"，本项目的评估终点是"行为改变"。引入从 RAG调用 → 后续行为窗口 → 攻击结果 的完整因果链评估，首次将"行为采纳度"定义为RAG系统的核心指标，而非相似度分数的补充。本次日志中 相似度0.817 但行为采纳度≈0 的案例，是这一创新点最直接的实证数据。

**创新点 II：渗透测试专项四层经验知识库**

通用的RAG反思系统无法区分"执行环境问题"和"情报认知问题"这两种性质完全不同的失败——前者换个环境就能成功，后者需要补充对目标的认知才能推进。本项目构建的五维失败根因分类法（ENV/INV/DEF/INT/EFF）和四层知识模型协同工作：五维分类法决定"这次失败应该提取哪类经验"，四层知识模型决定"这类经验应该以什么形式存储"。例如INT-WRONG_TOPOLOGY类失败提取为Metacognitive层的决策规则（如"loopback场景的Shell获取策略"），而ENV-BINARY_MISSING类失败提取为Procedural负向层的环境预检条目。这种分类-存储的双层对应关系，是通用RAG反思系统在架构层面无法实现的。

**创新点 III：渐进式自进化的异步反思闭环**

反思不干扰主流程（异步），反思结果以结构化经验条目形式沉淀（经验库），经验库成熟后回流为高优先级检索语料（闭环）。与现有SELF-RAG等在推理时做反思（realtime）不同，本方案的反思发生在任务结束后，以"知识进化"而非"实时纠偏"作为提升方式，对Agent主流程零侵入。

### 5.3 与 RAGFlow 现有功能的关系

RAGFlow 当前已有的记忆功能本质是会话级上下文记忆，即在单次对话内保持上下文连贯。本项目的反思机制与其的本质区别在于：

RAGFlow现有记忆解决的是"这次对话说过什么"，本项目解决的是"过去所有任务的经验如何影响未来任务的决策"，是跨任务、跨会话的经验沉淀。这两者在架构层面是互补关系，而非竞争关系，论文中可以明确说明本项目是在RAGFlow记忆功能之上构建的上层反思层。

| 字段           | 类型     | 说明                       |
| -------------- | -------- | -------------------------- |
| tool_name      | string   | 函数名，原样保留，不做分类 |
| call_args      | dict     | 参数字典，原样保留         |
| call_timestamp | datetime | 时间戳                     |

##### 2.2 结果描述（Result Descriptor）

result_fields 原样保留所有 tool_result 字段。

**可靠字段（规则直接读取，无需 LLM）**：

| 字段        | 类型   | 说明                                             |
| ----------- | ------ | ------------------------------------------------ |
| return_code | int    | 0 通常成功，非 0 通常失败，127 固定 = 命令不存在 |
| timed_out   | bool   | 直接读取，100% 可靠                              |
| success     | bool   | 工具自己报告，部分工具有此字段                   |
| stderr_raw  | string | 原始错误输出                                     |
| stdout_raw  | string | 原始标准输出                                     |

> **注意**：这里不做任何成功/失败信号词匹配，原样保留，交给 Layer 1 的语义理解层处理。

##### 2.3 行为归类（Action Category）——基于结构维度，不基于工具名

这是 Layer 0 唯一需要做的"分类"，分类依据只有两个来源：

**来源 A：结构化专用工具调用（STRUCTURED_TOOL_CALL）**

- tool_name ≠ generic_linux_command 的所有调用
- tool_name 和 call_args 原样保留，不做任何语义解读
- Layer 1 负责判断该工具的功能维度和行为类型
- 不枚举工具名，任何新工具自动归入此类

**来源 B：通用命令执行（GENERIC_COMMAND_CALL）**

- tool_name = generic_linux_command 或等价通用执行工具的所有调用
- 提取 program_name（command 字符串的首个可执行程序名，纯文本提取）
- command 字符串原样保留，不做子类型分类
- program_name 和完整 command 均交给 Layer 1 做语义理解

> **设计原因**：Layer 0 不知道也不应该知道 nmap 是侦察工具、hydra 是暴破工具，这些都是语义知识，属于 Layer 1 的职责。

**脚本写入事件（code_write）单独处理**：

- 识别 execute_code 或等价调用
- 原样保留 `{code, language, filename, timeout}` 全部字段
- 通过 filename/标识符 建立与后续执行该脚本的调用之间的关联
- filename 的语义（属于哪类攻击）交给 Layer 1 判断，Layer 0 不做解读
- 标注：code_write_event，关联后续 script_execute_event

##### 2.4 RAG 关联标记（仅做结构关联，不做语义评判）

- 识别 make_kb_search（或等价的知识检索调用）
- 记录 RAG 查询内容和时间戳
- 在时间窗口内（N=3 步）建立 RAG→行为 的时间关联
- **字段名**：`has_rag_context: true/false`（仅表示时间窗口内是否有 RAG 调用，不判断是否采纳）

##### 2.5 元数据提取

- session_id / start_time / end_time
- cost.total_cost（如有）
- timing_metrics（如有）
- 渗透目标（从首条含有效信息的 user_message 提取，不做解析）

------

#### Step 3：序列构建

将所有 Turn 按时间顺序排列，构成 TurnSequence。标记 execute_code 事件与后续 generic_linux_command 中执行对应脚本的关联关系（通过文件名匹配，与工具名无关）。

------

**Layer 0 输出**：结构化但语义中性的 TurnSequence，包含所有原始字段、时间顺序、RAG 时间关联。

不包含任何：阶段判断、失败类型、成功/失败标注。

------

### Layer 1：事件语义理解层

**输入**：Layer 0 输出的 Turn 对象序列

------

**输出**：

#### ① 全量 tool_call 事件的失败根因归类

无论是否有 RAG 支撑，所有失败事件均覆盖。归类体系采用**五维根因框架（ENV/INV/DEF/INT/EFF）**。

**规则层（先行，确定性处理）**：

| 条件              | 归类结果                      | 说明                       |
| ----------------- | ----------------------------- | -------------------------- |
| return_code = 127 | ENV-BINARY_MISSING            | shell 通用约定，命令不存在 |
| timed_out = true  | ENV-TIMEOUT                   | 工具自报告字段，直接读取   |
| return_code = 126 | ENV-EXECUTE_PERMISSION_DENIED | 无执行权限                 |

**LLM 层（兜底，处理其余所有失败情况）**：

基于五维框架判定，输出结构：

```json
{
  "dimension": "ENV / INV / DEF / INT / EFF",
  "sub_dimension": "具体子维度标签",
  "evidence": "判断依据（引用输出内容中的关键词/字段）",
  "remediation_hint": "下次应如何避免"
}
```

**五维根因框架定义**：

| 维度            | 含义                                   | 特征                     | 典型子维度                                                   |
| --------------- | -------------------------------------- | ------------------------ | ------------------------------------------------------------ |
| ENV（执行环境） | 失败原因在执行环境本身，与攻击策略无关 | 换正确配置的环境就能成功 | BINARY_MISSING / DEPENDENCY_MISSING / PERMISSION / TIMEOUT / RESOURCE |
| INV（调用方式） | 工具/命令调用方式本身有误，与目标无关  | 修正调用方式就能成功     | WRONG_ARGS / WRONG_SYNTAX / WRONG_SEQUENCE / WRONG_CONFIG    |
| DEF（目标防御） | 目标存在主动或被动防御机制             | 绕过防御或换方法才能成功 | AUTHENTICATION / AUTHORIZATION / ACTIVE_BLOCKING / RATE_LIMITING / PATCHED / MONITORING_DETECT |
| INT（情报认知） | 对目标或环境的认知不准确/不完整        | 补充正确信息后才能成功   | WRONG_VERSION / WRONG_TOPOLOGY / INCOMPLETE_RECON / WRONG_ASSUMPTION / MISSING_CONTEXT |
| EFF（执行效果） | 命令/漏洞被触发但效果与预期不符        | 需要调整后渗透策略       | BLIND_EXECUTION / PARTIAL_SUCCESS / OUTPUT_LOST / UNSTABLE   |

------

#### ② 全量 tool_call 事件的行为功能分类

分类体系：**Kill Chain 对齐的七维行为功能框架**（与 CAI 工具分类体系保持一致，基于安全杀伤链）。

由 LLM 基于 tool_name + call_args + 上下文判定，不依赖工具名枚举。

| 维度                                               | 描述                                            | LLM 判定依据                                   |
| -------------------------------------------------- | ----------------------------------------------- | ---------------------------------------------- |
| **侦察与武器化**（Reconnaissance & Weaponization） | 获取目标信息、制备攻击载荷的行为                | 行为目标是否为"获取信息"或"制备攻击载荷"       |
| **漏洞利用**（Exploitation）                       | 主动触发漏洞获取未授权初始访问权限的行为        | 是否在尝试获取初始立足点（first foothold）     |
| **权限提升**（Escalation）                         | 在已有初始访问基础上提升权限的行为              | 是否在当前权限基础上尝试获取更高权限           |
| **横向移动**（Lateral Movement）                   | 在已有立足点基础上向其他系统/服务扩展访问的行为 | 是否在尝试从当前立足点访问其他目标             |
| **数据渗出**（Exfiltration）                       | 将目标数据传输到攻击者控制端的行为              | 是否在将数据从目标传出到攻击者端               |
| **命令与控制**（Command & Control）                | 建立持久化访问通道和维持控制的行为              | 是否在建立可持续控制通道（而非一次性命令执行） |
| **环境准备**（Environment Preparation）★           | 为攻击做准备但不直接操作目标的行为              | 操作对象是攻击者本地环境，而非目标系统         |

> **注意**：环境准备维度不在 CAI 原始 Kill Chain 中，是反思系统新增的扩展维度，用于捕获 Agent 的"自我准备"行为，对环境类缺口检测有独立价值。

------

#### ③ 成功语义判断

LLM 依据 session 目标动态判定是否达成攻击目标，输出：

```json
{
  "is_success": true/false,
  "success_type": "RCE_CONFIRMED / CREDENTIAL_OBTAINED / DATA_ACCESSED / FLAG_FOUND / PARTIAL",
  "evidence": "判断依据",
  "session_goal_achieved": true/false
}
```

------

#### ④ RAG 行为因果标注

仅针对 `has_rag_context=true` 的 Turn。

输出：`adopted / ignored / partial` + LLM 判定依据

**采纳度三级定义**：

| 级别                                      | 定义                                                         | 权重 |
| ----------------------------------------- | ------------------------------------------------------------ | ---- |
| Level 3：直接引用（Direct Adoption）      | 后续 tool_call 的参数/代码中包含 RAG 内容中的具体字符串（命令/代码片段/路径/端点） | 1.0  |
| Level 2：参考改写（Informed Adaptation）  | 后续行为的技术路线与 RAG 内容一致，但具体命令/代码由 Agent 自行生成 | 0.6  |
| Level 1：思路启发（Conceptual Influence） | RAG 内容影响了攻击方向选择，但具体技术路线不同               | 0.3  |
| Level 0：未采纳（Ignored）                | 后续行为与 RAG 内容无任何关联                                | 0.0  |

**行为采纳度（BAR）计算**：

```
BAR = Σ(各RAG调用的加权采纳级别) / RAG调用总次数
取值范围：[0.0, 1.0]
```

------

#### ⑤ has_rag_context=false 事件的知识缺口信号标记

区分"合理自主执行"与"知识库缺口暴露"两种情况，输出缺口信号及优先级（P0/P1/P2）。

**缺口分级规则**（基于失败根因维度动态生成，不预先枚举具体内容）：

| 级别               | 触发条件                                 | 典型场景                         |
| ------------------ | ---------------------------------------- | -------------------------------- |
| **P0**（立即触发） | 失败根因维度为 INT 且 result=failure     | 情报认知错误直接导致攻击无法推进 |
| **P1**（每日批量） | 失败根因维度为 ENV 或 INV 且无知识库覆盖 | 工具使用方式/存在性知识缺失      |
| **P2**（周期触发） | 定时任务，与具体失败无关                 | 安全社区知识库常规更新           |

------

**Layer 1 核心逻辑**：

- 确定性规则（return_code=127/126，timed_out）优先处理
- LLM 作为主力分类器，处理所有需要语义理解的判断
- few-shot 示例覆盖五维框架的典型场景（而非枚举具体工具的具体失败）

------

### Layer 2：经验提取与组织层

**输入**：Layer 1 输出的语义标注后事件流

**输出**：四层结构化经验条目（raw 状态，待 Layer 3 成熟）

| 层级              | 内容                                          |
| ----------------- | --------------------------------------------- |
| Factual 层        | 有效端口/版本/路径/payload 等可复现事实       |
| Conceptual 层     | 多次验证后可泛化的攻击模式与边界条件          |
| Procedural 正向层 | 成功命令序列 + exploit 代码模板（含前置条件） |
| Procedural 负向层 | 错误命令格式 + 失败原因 + 正确做法提示        |
| Metacognitive 层  | 决策反思条目（阶段级 + 会话级）               |

**核心逻辑**：LLM 生成结构化内容 + 规则引擎控制写入

- LLM 负责内容生成（按四层模板格式化输出）
- 规则引擎负责去重检测（向量相似度）
- 规则引擎负责写入触发条件判断（置信度阈值、去重阈值）
- execute_code 成功代码 → 直接写入 Procedural 正向层（高优先级）
- 负向命令格式 → 直接写入 Procedural 负向层（触发：任意失败事件）

------

**所有经验条目 metadata 必填字段**：

```json
{
  "knowledge_layer": "Factual / Conceptual / Procedural_pos / Procedural_neg / Metacognitive",
  "attack_phase": "RECON_WEAPONIZATION / EXPLOITATION / ESCALATION / LATERAL_MOVEMENT / EXFILTRATION / COMMAND_CONTROL / ENV_PREPARATION",
  "target_service": "WebLogic / CouchDB / Apache Druid 等",
  "cve_id": "CVE-XXXX-XXXXX（如适用）",
  "maturity": "raw / validated / consolidated",
  "confidence_score": 0.0~1.0,
  "session_source": "session_id",
  "root_cause_dimension": "ENV / INV / DEF / INT / EFF（失败类经验）",
  "applicable_constraints": {
    "service_versions": ["10.3.x", "12.1.x"],
    "os_type": ["linux", "windows"],
    "network_topology": ["loopback", "intranet", "internet"],
    "precondition_services": ["未认证CouchDB", "admin party模式"],
    "known_inapplicable": ["已打补丁版本", "开启了认证的实例"]
  },
  "defense_bypass": {
    "defense_type": "WAF / IDS / EDR / 速率限制（如适用）",
    "bypass_technique": "具体绕过技术描述"
  }
}
```

**检索时的匹配逻辑**：Agent 在检索经验时，将当前 session 的环境上下文（target IP 类型/服务版本/已知配置）与 applicable_constraints 做匹配，不满足约束的经验条目降权或过滤，避免误用。

------

### Layer 3：经验成熟与融合层

**输入**：经验库中的 raw 状态条目

**输出**：validated / consolidated 状态的高质量经验条目

**核心逻辑（三条并行机制）**：

#### ① 自动融合（规则触发 + LLM 执行）

触发条件：同类攻击链经验条目 ≥ 3 条

**"同类"判定标准**（优先级从高到低）：

1. cve_id 相同 → 直接认定为同类
2. target_service + attack_phase 均相同 → 认定为同类
3. knowledge_layer + root_cause_dimension 均相同 → 认定为潜在同类，由 LLM 确认

**融合类型**：

| 类型       | 触发                     | 输出                     |
| ---------- | ------------------------ | ------------------------ |
| 横向融合   | 同漏洞不同环境的正向经验 | 版本-端点映射 Conceptual |
| 纵向融合   | 同漏洞正向 + 负向经验    | 完整攻击手册 Procedural  |
| 通用化提升 | 多框架验证的攻击模式     | 普适攻击模式经验         |

LLM 执行融合，规则引擎更新 maturity 字段。

#### ② 人工复核队列

- **触发条件**：置信度处于临界区间 [0.5, 0.7]
- **处理方式**：标记进入人工审核队列，审核通过后升级为 validated

#### ③ 自动置信度升级

- **触发条件**：经验条目被引用且对应阶段 result=success
- **处理方式**：每次成功引用 +0.05 置信度，上限 1.0；每次被引用但 result=failure，-0.03；超过 12 个月未被成功引用，每月 -0.02（时效衰减）

------

### Layer 4：知识库维护层

**输入**：

- Layer 1 输出的知识缺口信号（P0/P1/P2 分级）
- 经验库中时效衰减信号（超期未成功引用的条目）
- 负向经验中的冲突信号（新经验与知识库现有内容矛盾）

**输出**：主知识库的增 / 删 / 更新操作

**核心逻辑**：

#### ① 爬虫调度（三级优先队列）

| 级别 | 触发条件                                            | 处理时机 |
| ---- | --------------------------------------------------- | -------- |
| P0   | `has_rag_context=false` 且 failure 且根因维度为 INT | 立即触发 |
| P1   | 失败根因维度为 ENV 或 INV 且无知识库覆盖            | 每日批量 |
| P2   | 定时任务（与具体 session 无关）                     | 周期触发 |

#### ② 内容质量过滤

爬取内容需满足：包含可执行命令序列 或 PoC 代码 或 CVE 详情，不满足条件的内容不写入主知识库。

#### ③ 冲突检测与清理

- 新增负向经验 → 自动检索知识库中推荐该错误做法的文档 → 标记/降权/删除
- 版本修补信息 → 自动扫描仍推荐该 CVE 的文档 → 标记/降权/删除

------

### Layer 5：经验回流层（闭环关键）

**输入**：maturity=consolidated 且 confidence_score > 0.8 的成熟经验条目

**输出**：写回主知识库，作为最高检索优先级语料

**核心逻辑**：定期批量同步（非实时，保证稳定性，建议每周一次）

**回流策略**（按知识层级分类写入）：

| 知识层级          | 回流形式                   | 内容                                                  |
| ----------------- | -------------------------- | ----------------------------------------------------- |
| Factual 层        | 靶场环境指纹库             | 有效端口/版本/路径/payload 的快查表                   |
| Procedural 正向层 | 渗透测试操作手册           | exploit 代码模板、命令序列、前置条件检查清单          |
| Procedural 负向层 | 常见错误规避指南           | 错误命令格式 + 正确做法，Agent 调用前参考             |
| Metacognitive 层  | 决策规则库（最高检索权重） | 渗透测试专项决策原则，影响 Agent 下次任务的决策优先级 |

**回流后效果**：下次 Agent 检索时，经验库回流内容优先级高于原始爬虫文档，实现"过去任务经验 → 当前任务决策支撑"的跨会话知识进化闭环。

## 七、度量体系（支撑顶会论文的实验设计）

评估指标设计是顶会的核心审查点，本项目需要建立两套度量：

**A. 系统度量（衡量反思机制本身的质量）**

RAG行为采纳率的变化趋势：随着经验库积累，Agent对RAG内容的采纳度是否提升。知识缺口填补率：P0级缺口在下次同类任务前是否被覆盖。经验库精度：consolidated经验条目在后续任务中的正确引导率。

**B. 任务度量（衡量反思对渗透测试结果的提升）**

相同靶机环境下，有/无反思经验库的Agent在以下指标上的对比：成功获得Shell的轮次数（越少越好）、工具调用失败率（越低越好）、决策路径最优性（与专家预设最优路径的偏差）。

------

## 八、论文定位建议

综合以上，本项目的顶会投稿方向建议定位在以下两类：

**主投方向：** ACL/EMNLP（NLP顶会，RAG方向）或 USENIX Security/IEEE S&P（安全顶会，LLM+安全方向）。前者强调行为因果评估这一NLP创新，后者强调渗透测试自动化提升这一安全应用价值。

**论文框架结构：**

```
Abstract: 行为因果RAG评估 + 渗透测试专项经验库 → 攻击成功率提升X%
Introduction: RAG在安全LLM Agent中的应用困境（相似度≠有用性）
Related Work: SELF-RAG/CRAG/安全LLM Agent对比
Method: 四层知识模型 + 行为因果评估 + 异步反思闭环
Experiment: 多靶场环境对比实验 + 消融实验（去掉各创新点的影响）
Conclusion: 局限性（数据量/靶场覆盖范围）+ 未来方向（Embedding微调）
```
