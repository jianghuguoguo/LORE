# ***\*§1  问题定义与研究动机\****

## ***\*1.1  背景：自主渗透测试智能体的知识瓶颈\****

自主渗透测试（Autonomous Penetration Testing, AutoPT）是当前安全领域的前沿方向。以 PentestGPT、HackingBuddyGPT 等系统为代表的智能体，已经能够在受控靶机环境下完成基础的信息收集、漏洞识别和利用。然而，这些系统普遍面临一个核心瓶颈：

知识不能跨会话积累。每次渗透会话结束后，智能体获得的所有经验——哪些漏洞有效、哪些路径被防护、哪些CVE对当前版本无效——全部丢失。下一次面对同类目标时，智能体仍然从零开始探索，重复同样的错误。

这与人类渗透测试专家的成长模式截然相反。经验丰富的渗透测试工程师能够建立高质量的个人知识库，在面对新目标时快速调取历史经验，大幅缩短测试周期。Layer2 的经验提取系统正是为了解决这一问题而设计的——它从每次渗透会话中提取结构化经验，存储为 FACTUAL、PROCEDURAL_NEG、PROCEDURAL_POS、METACOGNITIVE 和 CONCEPTUAL 五层知识。

## ***\*1.2  新问题：多会话经验的融合挑战\****

当经验库累积了来自多个会话的原始经验后，新的问题随之出现：

**•** ***\*冗余膨胀：\****同一服务（如 Oracle WebLogic Server）被测试了 4 次，每次都生成了相似的 RECON_BEFORE_EXPLOIT 规则，导致检索时该规则被命中 4 倍，人为放大其权重。

**•** ***\*矛盾共存：\****两次会话中，一次发现 CVE-2017-10271 已被修补（HTTP 500），另一次在不同版本的目标上成功利用了相同 CVE。这两条经验在库中同时存在，Agent 检索时无从判断哪条适用。

**•** ***\*置信度固化：\****原始经验（raw maturity）的置信度为 0.3。即使同一规律在 5 个独立会话中被重复验证，系统无法自动将其升级为高置信度（validated）知识。

**•** ***\*版本约束丢失：\****单条经验记录了 CVE-2017-10271 在版本 10.3.6.0 上失效，但没有声明对其他版本的适用性。融合时需要推断出版本维度的约束范围。

**•** ***\*时序失效：\****较早期发现的 CVE 利用路径可能因为新补丁发布而失效，但系统缺乏根据时间维度降权的机制。

 

| ***\*核心研究问题（Research Questions）\****                 |
| ------------------------------------------------------------ |
| ***\*RQ1:\**** 如何在保留每条经验独特信息的前提下，将同主题的多条原始经验融合为一条高置信度的复合经验？ |
| ***\*RQ2:\**** 当同类经验存在矛盾时（同一CVE在不同条件下成功/失败），如何表示和利用这种条件化矛盾？ |
| ***\*RQ3:\**** 如何设计可量化的置信度演化机制，使经验的可信度随着证据积累自动提升？ |
| ***\*RQ4:\**** 如何在知识融合的同时保证融合过程可逆，支持审计和回滚？ |

 

 

# ***\*§2  相关工作分析\****

本框架与六个相关研究方向存在关联，但均存在根本性的方法论差异：

## ***\*2.1  知识图谱融合与实体对齐\****

典型工作：OpenEA（KDD'20）、PARIS（VLDB）、MultiKE（IJCAI'19）等。这些方法将知识表示为三元组 <subject, predicate, object>，通过实体嵌入或属性匹配实现跨图谱对齐。

局限性：渗透测试经验的核心结构是自然语言+条件逻辑（IF-THEN-NOT），而不是三元组。CVE 的语义相似性依赖安全领域本体（如 CVE 编号、CVSS 评分、受影响版本范围），通用知识图谱方法无法处理这种领域特异性。

## ***\*2.2  持续学习（Continual Learning）\****

典型工作：EWC（PNAS'17）、iCaRL（CVPR'17）、Replay-based CL（NeurIPS'19）。这些方法关注神经网络在顺序任务上避免灾难性遗忘。

局限性：持续学习假设知识以神经网络参数形式存储，而我们的经验库是显式的结构化符号知识。此外，持续学习不涉及跨经验的主动融合——它只是保留旧知识，不会将两条经验合并为一条。

## ***\*2.3  RAG 知识库更新\****

典型工作：REALM（ICML'20）、RAG（NeurIPS'20）、Self-RAG（ICLR'24）。

局限性：RAG 系列工作假设检索库是相对静态的文档集合，知识更新等同于添加新文档。没有工作处理如何将库中的两个语义相似文档合并为一个权威版本，也没有处理文档之间的矛盾关系。

## ***\*2.4  经验回放（Experience Replay）in RL\****

典型工作：DQN、HER（NIPS'17）、PER（ICLR'16）优先经验回放。

局限性：RL 中的经验是 <state, action, reward, next_state> 四元组，融合等同于采样和加权更新神经网络参数。我们的经验是结构化决策规则，融合的目标是提炼可解释的 IF-THEN 规则，而非更新参数。

## ***\*2.5  自动化渗透测试（AutoPT）\****

典型工作：AutoAttacker、PentestGPT（USENIX'24）、HackingBuddyGPT（LLM4Shell）、PENTESTBENCH。这些工作关注智能体的渗透能力，但均未涉及跨会话经验的积累和融合。PentestGPT 虽然使用 RAG，但其知识库是静态的 CTF 解题文档，不是动态生成和融合的经验。

## ***\*2.6  知识蒸馏与合并\****

典型工作：模型合并（Model Merging）、知识蒸馏（Hinton et al.）。

局限性：这些工作关注神经网络层面的知识迁移，不处理符号规则的合并问题。

 

| ***\*研究空白总结（Research Gap）\****                       |
| ------------------------------------------------------------ |
| 目前没有任何工作研究如何从多次渗透测试会话中提炼可泛化的、带置信度演化机制的结构化决策规则，并支持规则的条件化表示、冲突检测和生命周期管理。这是一个完全未被填补的研究空白。 |

 

 

# ***\*§3  核心理论基础\****

## ***\*3.1  条件化经验（Conditioned Experience）\****

传统经验表示将经验视为无条件的规则：「CVE-2017-10271 对 Oracle WebLogic 有效」。这种表示对矛盾无能为力。

本框架引入条件化经验的概念，将任意经验 E 表示为一个带条件约束的五元组：

 

| ***\*定义 3.1：条件化经验（Conditioned Experience, CE）\**** |
| ------------------------------------------------------------ |
| CE = ⟨K, C_pos, C_neg, W, T⟩ 其中：                          |
| ***\*K\****  ∈ KnowledgeBase：知识核（IF-THEN-NOT 规则或具体事实） |
| ***\*C_pos\****  = { c₁, c₂, ..., cₙ }：正适用条件集合（满足时规则有效） |
| ***\*C_neg\****  = { d₁, d₂, ..., dₘ }：负适用条件集合（满足时规则无效） |
| ***\*W\****  ∈ [0, 1]：证据权重（Evidence Weight，由置信度和成熟度决定） |
| ***\*T\****  ∈ Timestamp：创建时间（用于时效性衰减计算）     |

 

示例：

CE_1 = ⟨"CVE-2017-10271 XMLDecoder路径有效", {"WebLogic版本∈[10.3.x, 12.1.x]", "wls-wsat端点可达"}, {"返回HTTP 500"}, 0.75, 2026-01-15⟩

CE_2 = ⟨"CVE-2017-10271 XMLDecoder路径无效", {"WebLogic 10.3.6.0", "wls-wsat返回HTTP 500"}, {}, 0.85, 2026-02-20⟩

CE_1 和 CE_2 不再是矛盾的——它们表达了同一 CVE 在不同条件下的不同结论，各自有独立的适用范围。

## ***\*3.2  证据权重函数（Evidence Weight Function）\****

每条经验的可信度由以下三个维度决定：

 

| ***\*定义 3.2：证据权重计算\****                             |
| ------------------------------------------------------------ |
| W(E) = α × session_factor(E) × β × maturity_factor(E) × γ × outcome_factor(E) |
| ***\*session_factor\****  = session_bar_score ∈ [0, 1]（会话整体质量，越高越可信） |
| ***\*maturity_factor\**** = { raw: 1.0, validated: 2.0, consolidated: 3.5 }（成熟度乘数） |
| ***\*outcome_factor\****  = { success: 1.5, partial_success: 1.0, failure: 0.7 }（结果乘数） |
| ***\*α, β, γ\****     = 归一化系数，使 W(E) 归一到 [0, 1]    |

## ***\*3.3  贝叶斯置信度融合（Bayesian Confidence Fusion）\****

当 n 条相互独立的经验 {E₁, E₂, ..., Eₙ} 同时支持同一规律 K 时，融合后的置信度采用独立证据的贝叶斯更新：

 

| ***\*定义 3.3：贝叶斯置信度融合公式\****                     |
| ------------------------------------------------------------ |
| P(K \| E₁, E₂, ..., Eₙ) = 1 - ∏ᵢ (1 - P(K \| Eᵢ))            |
| 这等价于：多个独立证据共同支持同一假设时，合并置信度随证据数量快速收敛到 1。 |
| 示例：3条独立raw经验各自置信度 0.3 → 1-(1-0.3)³ = 1-0.343 = 0.657，已接近 validated 阈值 0.7。 |
| 若经验之间存在相关性（来自同一靶机），使用保守公式：         |
| P_correlated = P(K \| E₁) + W₂ × (1 - P(K \| E₁)) + W₃ × (1 - P_prev) × ... |

## ***\*3.4  知识生命周期（Knowledge Lifecycle）\****

经验的可信度随时间自然衰减（CVE 被修复、补丁发布），随新证据的到来自然增长。定义时效性衰减函数：

 

| ***\*定义 3.4：时效性衰减函数\****                           |
| ------------------------------------------------------------ |
| W_effective(E, t) = W(E) × exp(-λ × Δt)                      |
| ***\*Δt\**** = 从经验创建到当前时间（天数）                  |
| ***\*λ\****  = 衰减率，按安全域分类：                        |
| λ_CVE_exploit  = 0.005  (约140天半衰期，CVE利用相对稳定)     |
| λ_network_topo  = 0.002  (约350天半衰期，网络拓扑变化慢)     |
| λ_tool_usage   = 0.010  (约70天半衰期，工具版本更新快)       |
| λ_auth_bypass  = 0.020  (约35天半衰期，认证机制变化最快)     |
| 当 CVE 被标记为 "patched" 的新经验出现时，对应旧经验的 λ 临时加速（×5），触发快速降权。 |

 

 

# ***\*§4  XPEC 融合框架全貌\****

本框架命名为 XPEC（eXperience-based Penetration-test Experience Consolidation），是一个五阶段的持续知识演化管道。

 

| ***\*XPEC 框架：五阶段管道\****                              |
| ------------------------------------------------------------ |
| ***\*Phase 1 — Semantic Equivalence Clustering（SEC）\****：语义等价聚类 |
| 将来自不同会话的经验按语义相似性分组，找出「说同一件事」的经验集合。 |
|                                                              |
| ***\*Phase 2 — Evidence Weight Calculation（EWC）\****：证据权重计算 |
| 为每条经验计算权重 W(E)，综合考量会话质量、成熟度和结果类型。 |
|                                                              |
| ***\*Phase 3 — Rule Merge Engine（RME）\****：规则融合引擎   |
| 在每个等价组内，执行 IF/THEN/NOT/next_actions 的多数投票融合，产出复合经验。 |
|                                                              |
| ***\*Phase 4 — Bayesian Confidence Calibration（BCC）\****：贝叶斯置信度校准 |
| 基于贝叶斯公式计算融合后的置信度，决定是否升级 maturity 等级。 |
|                                                              |
| ***\*Phase 5 — Knowledge Lifecycle Management（KLM）\****：知识生命周期管理 |
| 执行时效性衰减、成熟度升级、软删除标记，维护经验库的健康状态。 |

 

整个管道的设计原则：

**•** ***\*非破坏性（Non-Destructive）：\****融合不删除任何原始经验，仅在此之上创建新的 consolidated 级别经验，并通过 merged_into 指针链接。

**•** ***\*可解释性（Explainable）：\****每条 consolidated 经验都携带 provenance（来源经验列表），Agent 可以随时追溯「这条规则是怎么得出的」。

**•** ***\*条件化（Conditioned）：\****融合后的经验显式携带适用条件和排除条件，不会生成过度泛化的规则。

**•** ***\*可逆性（Reversible）：\****通过 merged_into 指针回滚，支持人工审核后的撤销操作。

 

 

# ***\*§5  Phase-1：语义等价聚类（SEC）\****

## ***\*5.1  聚类维度设计\****

SEC 的核心挑战是：如何判断两条来自不同会话的经验「说的是同一件事」？单纯的向量相似度不够——两条关于不同 CVE 但描述同一类失败模式的经验，向量距离可能很近，但绝对不应该合并。

SEC 采用分层硬匹配（Hard Key）与软语义匹配（Soft Semantic）结合的策略：

 

| ***\*层级\**** | ***\*匹配方式\**** | ***\*匹配字段\****                                       | ***\*权重\**** | ***\*说明\****                  |
| -------------- | ------------------ | -------------------------------------------------------- | -------------- | ------------------------------- |
| L1 硬键        | 精确匹配           | knowledge_layer + target_service + failure_sub_dimension | 必要条件       | 三者全匹配才进入同一候选组      |
| L2 软键        | 关键字重叠         | CVE IDs 交集 ≥ 1，或 path/endpoint 重叠 ≥ 60%            | 强信号         | CVE 完全相同则直接分组          |
| L3 语义        | 余弦相似度         | decision_rule.IF 的嵌入向量                              | 辅助           | cosine(v_IF_i, v_IF_j) > τ=0.82 |
| L4 版本        | 范围约束           | target_version 是否在同一版本族（如 10.3.x）             | 排除条件       | 版本族不同则强制分入不同组      |

## ***\*5.2  安全领域语义本体（Security Semantic Ontology, SSO）\****

通用 embedding 模型不理解 CVE 语义。CVE-2017-10271 和「WebLogic XMLDecoder 反序列化」是同一概念，但在 text-embedding-ada-002 的向量空间中距离较大。为此，SEC 引入 SSO 进行规范化：

 

| ***\*SSO 规范化示例\****                                     |
| ------------------------------------------------------------ |
| ***\*CVE别名映射：\****"wls-wsat XMLDecoder漏洞" → CVE-2017-10271（来自 NVD 描述的别名索引） |
| ***\*服务名称规范化：\****"Oracle WebLogic" ≡ "WebLogic Server" ≡ "WLS"（来自厂商品牌别名库） |
| ***\*路径语义等价：\****"/wls-wsat/CoordinatorPortType" ≡ "/wls-wsat/RegistrationPortTypeRPC"（同类端点族） |
| ***\*漏洞类型归类：\****"XMLDecoder反序列化" ⊂ "Java反序列化漏洞" ⊂ "反序列化"（层次化归类树） |

SSO 的构建来源：NVD API（CVE 描述）、MITRE ATT&CK（战术、技术映射）、CVE MITRE 官方别名库、人工标注的渗透测试专业词汇表。

## ***\*5.3  聚类算法\****

在经过 SSO 规范化后，SEC 执行以下聚类流程：

1. 对所有经验按 L1 硬键分组，形成初始候选组集合 G = {g₁, g₂, ...}。
2. 在每个候选组内，以 CVE IDs 交集（L2）为边构建相似图 Graph(G)。
3. 对 Graph(G) 执行连通分量分析，每个连通分量为一个等价集 S。
4. 对等价集内 L3 语义相似度 < τ 的经验对，强制分入独立子集（防止过度合并）。
5. L4 版本约束：将不同版本族的经验分入带版本标签的子集 S_v。

聚类输出：{(S₁, version_constraint₁), (S₂, version_constraint₂), ...}，每个等价集有明确的版本适用范围。

 

 

# ***\*§6  Phase-2：证据权重计算（EWC）\****

## ***\*6.1  权重维度与计算\****

每条经验 E 的权重向量 w = (w_quality, w_maturity, w_outcome, w_coverage)：

 

| ***\*w_quality\****   | session_bar_score × confidence（综合评分×提取置信度，∈[0,1]） |
| --------------------- | ------------------------------------------------------------ |
| ***\*w_maturity\****  | raw=0.4, validated=0.7, consolidated=1.0（成熟度系数）       |
| ***\*w_outcome\****   | success=1.5, partial_success=1.0, failure=0.6（结果乘数）    |
| ***\*w_coverage\****  | len(cve_ids)/max_cve_per_layer × 0.3 + 0.7（CVE覆盖度加成）  |
| ***\*W(E) 最终值\**** | 归一化(w_quality × w_maturity × w_outcome × w_coverage)，压缩到[0.1, 1.0] |

## ***\*6.2  特殊情况处理\****

**•** ***\*零质量经验：\****session_bar_score=0 的会话（6个零输出session）对应的经验权重 W_min=0.05，保留但影响极小。

**•** ***\*成功session的额外加成：\****如果等价集 S 中包含 outcome=success 的经验，该经验的 w_outcome 额外乘以 1.5，成为融合时的主导证据。

**•** ***\*时效性折扣：\****W_effective(E, t) = W(E) × decay_factor(E.layer, Δt)，确保较老经验不会主导融合结果。

 

 

# ***\*§7  Phase-3：规则融合引擎（RME）\****

## ***\*7.1  各知识层的融合策略\****

不同知识层的融合逻辑不同，需要针对性设计：

 

### ***\*7.1.1  PROCEDURAL_NEG 融合（最复杂，优先级最高）\****

NEG 记录的是失败决策规则（IF-THEN-NOT + next_actions），是 Agent 最频繁检索的经验类型（占95条/171条），融合质量直接影响 Agent 决策效率。

| ***\*NEG 融合算法\****                                       |
| ------------------------------------------------------------ |
| ***\*Step 1 — IF 融合（最大公约数法）\****：                 |
| 对等价集 S = {E₁, E₂, ..., Eₙ} 的所有 IF 语句，识别公共触发条件（weighted intersection）： |
| IF_merged = argmax_{条件子集 C} Σᵢ W(Eᵢ) × ��[C ⊆ IF(Eᵢ)]    |
| 即：找到被最多高权重经验共同包含的条件子集，作为融合后的触发条件。 |
|                                                              |
| ***\*Step 2 — THEN 融合（加权多数投票）\****：               |
| 对所有 THEN 建议，按出现频率×权重排序，选取加权票数超过阈值 θ=0.4 的建议： |
| THEN_merged = { t \| Σ_{Eᵢ: t∈THEN(Eᵢ)} W(Eᵢ) / Σᵢ W(Eᵢ) ≥ θ } |
| 未达到阈值但权重排名前3的建议：降权保留，标记为 minority_opinion=true。 |
|                                                              |
| ***\*Step 3 — NOT 融合（并集策略）\****：                    |
| NOT（禁止重试的操作）采用并集：所有 NOT 建议都加入 merged NOT 列表，任一建议触发都应回避。 |
|                                                              |
| ***\*Step 4 — next_actions 融合\****：                       |
| 按步骤序号对齐，相似步骤取权重最高版本，差异步骤保留多个选项（标记为 alternatives）。 |

 

### ***\*7.1.2  FACTUAL(LLM) 融合（CVE 地图汇聚）\****

多个 FACTUAL_LLM 经验记录了同一服务在不同会话中的 CVE 利用尝试结果。融合目标是建立该服务的完整 CVE 状态地图：

**•** ***\*cve_attempted 合并：\****取并集，注明每个CVE的尝试次数和成功率。

**•** ***\*exploitation_results 合并：\****按 CVE 编号分组，对同一CVE的多次结果，按权重投票得出综合状态（patched/partial/exploited）。

**•** ***\*known_ineffective_vectors 合并：\****取并集，注明每个路径的失效验证次数（多次验证比单次更可靠）。

**•** ***\*target_version 融合：\****若多条记录版本不一致，保留所有版本并标记各自的 exploitation_status。

 

### ***\*7.1.3  METACOGNITIVE 融合（规则去重与提权）\****

META 层是跨 session 重复率最高的层（RECON_BEFORE_EXPLOIT 出现4次，FAILURE_PIVOT 出现3次）。融合策略：

**•** ***\*key_lessons 语义去重：\****对 IF 语句进行语义聚类，相似度 > 0.88 的 lesson 视为等价，取权重最高版本作为代表。

**•** ***\*具体化优先原则：\****若两条等价 lesson 中，一条是通用表述（"做好侦察"），另一条包含具体CVE（"WebLogic在exploit前先确认CVE-2017-10271补丁状态"），保留具体版本。

**•** ***\*服务专属vs通用规则分离：\****RECON_BEFORE_EXPLOIT 这类通用规则单独存为 universal_rule，不与服务专属规则混合。

 

### ***\*7.1.4  CONCEPTUAL 融合（规律升级）\****

CONCEPTUAL 是最高层次的抽象，来自单个 session 的 confidence=0.3。融合的目标是将多 session 验证的规律升级置信度：

**•** ***\*core_insight 融合：\****用 LLM 对多条等价 CONCEPTUAL 的 core_insight 进行语义合并，生成更完整的综合洞察。

**•** ***\*applicable_conditions 加强：\****多条经验共同验证的条件（positive/negative）增加标注频次，高频条件置信度更高。

**•** ***\*supporting_evidence 累积：\****汇聚所有源经验的支撑证据，形成更强的证据链。

## ***\*7.2  融合完整性保障\****

每条 consolidated 经验必须携带完整的 Provenance：

# ***\*§8  Phase-4：贝叶斯置信度校准（BCC）\****

## ***\*8.1  置信度计算\****

融合后经验的置信度由以下公式计算：

| ***\*BCC 核心公式\****                                       |
| ------------------------------------------------------------ |
| ***\*独立经验融合（等价集内来自不同靶机的经验）：\****       |
| P_fused = 1 - ∏ᵢ (1 - Pᵢ × W(Eᵢ))                            |
|                                                              |
| ***\*相关经验融合（等价集内来自同一靶机/相近配置的经验）：\**** |
| P_fused = P_base + Σᵢ (Pᵢ - P_base) × W(Eᵢ) × (1 - 相关系数 ρ) |
| 其中 ρ = 1 表示完全相关（同一靶机重复测试），ρ = 0 表示完全独立。 |

## ***\*8.2  成熟度升级阈值\****

| ***\*当前成熟度\**** | ***\*升级条件\****                   | ***\*升级后\**** | ***\*置信度变化\**** |
| -------------------- | ------------------------------------ | ---------------- | -------------------- |
| raw (≤3条证据)       | P_fused ≥ 0.60 且独立证据来源 n ≥ 2  | validated        | 重新计算BCC值        |
| validated (4-9条)    | P_fused ≥ 0.80 且 n ≥ 3 且无严重矛盾 | consolidated     | BCC × 1.1，上限1.0   |
| consolidated         | 出现 n ≥ 2 条强反例（weight > 0.7）  | 降级回 validated | BCC × 0.6            |
| 任意                 | 出现 success session 的直接反例      | 立即重新评估     | 强制 BCC ≥ 0.5 重算  |

 

 

# ***\*§9  Phase-5：知识生命周期管理（KLM）\****

## ***\*9.1  经验的五种生命状态\****

| ***\*状态\**** | ***\*描述\****                   | ***\*RAG可检索\**** | ***\*触发条件\****                    |
| -------------- | -------------------------------- | ------------------- | ------------------------------------- |
| active         | 正常可用，参与检索               | ✅ 全权重            | 初始创建                              |
| archived       | 已被更高质量经验覆盖，降权但保留 | ⚠️ 0.3×权重          | 被 consolidated 经验 merged_into 覆盖 |
| suspended      | 时效性衰减至阈值以下             | ❌ 不参与            | W_effective < 0.1                     |
| conflicted     | 存在高权重反例，等待人工确认     | ⚠️ 单独标注          | BCC 矛盾检测触发                      |
| deleted        | 已确认无效，软删除               | ❌ 从索引移除        | 人工确认或 Layer3 强制降权            |

## ***\*9.2  定期维护计划\****

**•** ***\*每次新 session 处理后（实时）：\****检查新经验是否与现有 consolidated 经验冲突，触发 BCC 重新校准。

**•** ***\*每积累 5 个新 session（批量）：\****运行完整 XPEC 管道，对所有等价集执行一轮融合，检查成熟度升级机会。

**•** ***\*每 30 天（定期）：\****执行时效性衰减计算，将 W_effective < 0.1 的经验标记为 suspended。检查 NVD 数据库是否有新的 CVE 补丁发布，触发相关经验的加速衰减。

**•** ***\*新靶机目标类型出现时（事件驱动）：\****为新目标软件创建新的 SSO 映射，重新索引所有相关经验。

## ***\*9.3  回滚机制\****

由于所有原始经验保留在 JSONL 文件中（通过 exp_id 可溯源），任何融合操作都可以回滚：

 

| ***\*回滚操作流程\****                                       |
| ------------------------------------------------------------ |
| ***\*场景：\****发现 consolidated 经验 exp_consolidated_001 的融合结果有误（融合了不应合并的经验）。 |
| 1. 查询 exp_consolidated_001.provenance.source_exp_ids，得到 3 条原始经验 IDs。 |
| 2. 在向量库中执行：delete(ids=["exp_consolidated_001"])。    |
| 3. 在 experience_registry 中将 3 条原始经验的 merged_into 重置为 null，status 重置为 active。 |
| 4. 原始经验自动重新进入检索索引，系统恢复到融合前状态。      |
| 总耗时：< 100ms。无数据损失。                                |

 

 

# ***\*§10  冲突检测与矛盾经验处理\****

## ***\*10.1  冲突的四种类型\****

| ***\*冲突类型\**** | ***\*示例\****                        | ***\*处理策略\****                                |
| ------------------ | ------------------------------------- | ------------------------------------------------- |
| 直接逻辑矛盾       | E₁: CVE-X有效；E₂: CVE-X无效          | 按版本约束分离为两条 conditioned experience       |
| 动作冲突           | E₁ THEN: 尝试路径A；E₂ NOT: 禁止路径A | 保留高权重方，低权重方标记为 conflicted 并降权    |
| 时序冲突           | 旧经验: CVE有效；新经验: CVE已打补丁  | 新经验自动覆盖，旧经验 status→archived，加速衰减  |
| 版本冲突           | E₁(v10.3): 有效；E₂(v12.1): 无效      | 按版本分组，在各自 conditioned 经验中表达，不合并 |

## ***\*10.2  矛盾评分（Contradiction Score）\****

对等价集 S 中的任意经验对 (Eᵢ, Eⱼ)，计算矛盾评分：

 

| ***\*矛盾评分公式\****                                       |
| ------------------------------------------------------------ |
| Contra(Eᵢ, Eⱼ) = w_logic × logic_conflict + w_action × action_conflict + w_outcome × outcome_diff |
| 其中 outcome_diff = \|success_flag(Eᵢ) - success_flag(Eⱼ)\|（1表示结果相反） |
| ***\*Contra > 0.6：\****强矛盾，强制条件化分离，不执行融合   |
| ***\*0.3 < Contra ≤ 0.6：\****弱矛盾，执行条件化融合，minority_opinion 保留 |
| ***\*Contra ≤ 0.3：\****非矛盾，正常融合流程                 |

 

 

# ***\*§11  Layer3 触发策略与调度机制\****

## ***\*11.1  触发条件\****

**•** ***\*批量触发（主触发器）：\****experience_registry 中新增经验数 ≥ 5 且距上次 Layer3 运行时间 ≥ 1小时。

**•** ***\*即时冲突触发：\****新经验的 contradiction_score(新经验, 现有consolidated经验) > 0.6，立即触发 Phase 1-4 的局部重算。

**•** ***\*人工触发：\****研究人员发现融合质量问题时手动触发，支持指定等价集范围。

**•** ***\*周期触发：\****每30天强制执行全库时效性衰减和成熟度审查，不依赖新经验积累。

## ***\*11.2  计算复杂度分析\****

设经验总数为 N，等价集数为 M，每个等价集平均大小为 k = N/M：

**•** ***\*SEC 聚类：\****O(N × log N)（排序 + 连通分量），硬键分组使实际复杂度远低于理论上界。

**•** ***\*EWC：\****O(N)，每条经验独立计算。

**•** ***\*RME：\****O(M × k²)，每个等价集内两两比较。实际上 k 通常 ≤ 10，复杂度可控。

**•** ***\*BCC + KLM：\****O(M)，每个等价集一次计算。

在当前规模（N≈171, M≈30）下，完整 Layer3 管道运行时间预计 < 30 秒（含 LLM 调用）。当 N 扩展到 10,000 时，预计 ≈ 5 分钟。

 

 

# ***\*§12  评估方法论与指标体系\****

## ***\*12.1  核心评估指标\****

顶会论文要求可量化、可重复的评估。本框架提出以下指标体系：

 

| ***\*指标\****                  | ***\*定义\****                              | ***\*测量方法\****                 | ***\*目标值\****   |
| ------------------------------- | ------------------------------------------- | ---------------------------------- | ------------------ |
| First Blood Rate（FBR）         | Agent首次面对新靶机时在N步内成功利用的比率  | 50个独立靶机实验，统计10步内成功率 | 基线30%→目标55%+   |
| Mean Steps to Compromise（MSC） | 平均到达第一个成功利用点的步骤数            | 成功会话的步骤中位数               | 基线25步→目标16步  |
| Knowledge Precision（KP）       | 被Agent使用的经验中产生正确决策的比率       | 人工标注Agent决策质量              | 基线50%→目标75%+   |
| Fusion Soundness（FS）          | 融合后的consolidated经验中无错误矛盾的比率  | 人工审计100条consolidated经验      | > 90%              |
| Recall Completeness（RC）       | Agent检索到的经验中覆盖该场景有效知识的比率 | 对比人工专家标注的「应检索到」列表 | > 70%              |
| Layer3 Compression Ratio        | 融合后经验库大小/原始经验库大小             | 直接计算                           | 目标0.6（压缩40%） |

## ***\*12.2  消融实验设计\****

为证明每个 Phase 的必要性，设计以下消融实验：

| ***\*消融配置\**** | ***\*移除的组件\****                   | ***\*期望观察到的性能下降\****          |
| ------------------ | -------------------------------------- | --------------------------------------- |
| w/o SEC            | 无语义等价聚类，所有经验独立存储       | KP下降（冗余命中干扰）、FBR下降         |
| w/o SSO            | SEC中不使用安全本体，仅用通用embedding | 融合质量下降，CVE语义等价识别失败       |
| w/o BCC            | 置信度固定不更新，无成熟度升级         | Agent无法识别高置信度知识，决策质量下降 |
| w/o KLM            | 无时效性衰减，旧经验永久有效           | 过时CVE知识干扰新场景决策               |
| w/o Conditioned    | 矛盾经验直接合并（取高权重方）         | FS下降，矛盾经验导致错误决策            |
| Full XPEC          | 完整框架                               | 所有指标最优                            |
